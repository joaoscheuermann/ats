# Decision Log: ats-resume-optimizer-cli

### 2026-04-22 14:35 - Architect

**Context**: Designing the first implementation of the ATS Resume Optimizer CLI. Spec is comprehensive but left asset delivery, token pre-flight handling, crate layout, log sink/format, secrets policy, binary name, and PDF styling open. An `output/` folder for the final PDF was also added during architecture review.
**Decision**: Adopt a multi-crate Rust layout under `packages/*` (`ats-core`, `ats-scrape`, `ats-llm`, `ats-pdf`, `ats-cli`) with port traits in `ats-core` and all runtime concretions (`chromiumoxide`, `reqwest`, `markdown2pdf`) confined to adapter crates. Embed the frozen resume template, YAML schema, job-posting and keyword `response_format` JSON blocks, and all three LLM prompts via `include_str!` in `ats-core`. Config is read from `config.json` next to the binary with no env overrides and no defaults. Logs stream as JSON lines to stderr by default with a `--log-format=pretty` override. Every subcommand invocation (`render`, `scrape`, `keywords`, `optimize`, `pdf`, `run`) creates its own `runs/<ts>_<command>[_<slug>]/` folder containing `run.json` and, when LLM calls occur, `llm-audit.jsonl`. The `run` subcommand additionally copies the final PDF into `<binary_dir>/output/`. Amend AC-3.5 to **AC-3.5″**: do not tokenize locally and do not call `/models`; rely on OpenRouter's `usage` in each response and classify any context-length error as `llm/context-exceeded` (exit 5). Exit codes locked: `0 ok / 1 other / 2 config / 3 yaml / 4 scrape / 5 llm / 6 schema / 7 pdf`. Remove `pdf.style` from `config.json` for this version.
**Rationale**: The multi-crate split satisfies SRP and DIP, keeps `ats-core` testable without network or a browser, and matches the human's explicit preference for strongest boundaries. Embedding assets gives a single source of truth that participates in the AC-1.5 cache hash and honours NFC-4 "verbatim". Per-invocation audit folders honour NFC-19 consistently across all subcommands. The output folder makes the final deliverable trivial to find without wading through run intermediates. Dropping local tokenization matches the human's direction and eliminates a brittle heuristic layer in favour of the provider's authoritative usage reporting.
**Alternatives considered**: Single crate (simpler, fewer manifests, weaker boundaries — rejected per human choice); two-crate split (core + cli — rejected for the same reason); literal AC-3.5 with `tiktoken-rs` pre-flight counting (rejected per human direction); char-based heuristic token counting (rejected — unreliable); `async-std` runtime (rejected — chromiumoxide/reqwest are tokio-native); `ureq` blocking HTTP (rejected — forces spawn_blocking for LLM calls); bundling a Chromium binary (rejected — size and per-OS complexity; rely on system Chrome/Chromium detection); keeping `pdf.style` now (rejected — YAGNI per human); env-var secret overrides (rejected per human).

### 2026-04-22 14:40 - Decomposer

**Context**: Breaking the approved architecture for ats-resume-optimizer-cli into incremental, vertically sliced efforts that each produce an observable CLI behaviour. Each effort must stand up a subcommand the human can run end-to-end before the next effort begins.
**Decision**: Produced seven ordered efforts under `efforts/` — (01) scaffold workspace and CLI skeleton with clap, config loader, FsLayout, embedded assets, error taxonomy, logging, per-invocation audit folder; (02) `ats render` with YAML parsing, JSON Schema validation, hand-rolled Markdown rendering, and SHA-256 cache; (03) `ats keywords` end-to-end, which also lands the entire `ats-llm` OpenRouter client, transient retry, audit sink, schema-validation loop, and token-usage reporting; (04) `ats scrape` with `chromiumoxide`, reachability probe, and AC-2.3/AC-2.5 error classification, plus the scrape→Markdown LLM call; (05) `ats optimize` with the density metric and warning bands; (06) `ats pdf` via `markdown2pdf`; (07) `ats run` orchestrator that composes all stages in-process, materialises the run folder after the scrape slug is known, copies the final PDF into `output/`, and aggregates token usage.
**Rationale**: Keywords before scrape keeps the first network-touching slice free of browser dependencies, so LLM retry/audit/error classification can be proven before adding Chromium. Each subsequent effort adds one new observable subcommand that pipes into the previous ones (`ats scrape | ats keywords`, `ats render | ats optimize`, `... | ats pdf`), giving the human a working tool after every effort. The final `run` orchestrator reuses every stage unchanged, so late-stage risk is limited to orchestration and artefact materialisation. Every effort's Verification Criteria includes a run/observe step (release binary invocation against real or mocked services), not just unit tests.
**Alternatives considered**: Scrape before keywords (rejected — forces chromium integration into the first networked slice); splitting US-3 into separate "LLM client" and "keywords stage" efforts (rejected — would leave the LLM client with no observable subcommand); bundling PDF into the `run` effort (rejected — hides a real failure surface behind orchestration); a horizontal split (all ports first, then all stages) (rejected per decomposer philosophy — yields nothing runnable until very late).

### 2026-04-22 16:12 - Executor

**Context**: Post-completion review of Effort 02 surfaced a portability hazard: `hash_key` folds `RESUME_TEMPLATE.as_bytes()` into the cache key, but the compiled bytes carry CRLF on Windows checkouts (repo has `core.autocrlf=true`) and LF on Linux checkouts, so the same YAML would produce different cache keys on different machines.
**Decision**: Normalised line endings inside `hash_key` itself by stripping every `\r` byte from both inputs (and from the template bytes via a `pub(crate)` `hash_key_with_template`) before feeding SHA-256. Cached file contents on disk remain exactly what `render_baseline` produces (LF-only) — only the key is normalised. Added three tests: `hash_key_is_stable_across_crlf_vs_lf_yaml`, `hash_key_changes_when_non_newline_bytes_change`, `hash_key_is_stable_if_template_has_crlf`. One-time cache invalidation accepted (any cache files emitted before this fix will orphan under their old hex).
**Rationale**: Matches the user's explicit preference ("normalise in the hash function itself — cache survives per-machine regardless of checkout") and keeps the fix off the filesystem layer (no `.gitattributes` needed, no re-normalisation churn). 79 workspace tests pass (up from 76 — the three new tests); `cargo clippy --workspace --all-targets -- -D warnings` clean.
**Alternatives considered**: (a) Add `.gitattributes` forcing `packages/ats-core/assets/** text eol=lf` — rejected by the user because it would re-normalise the checked-in template and globally invalidate caches at first pull. (c) Leave it and defer to distribution — rejected as a known portability footgun.

### 2026-04-22 22:30 - Executor

**Context**: Efforts 06 ("Implement ats pdf via markdown2pdf") and 07 ("Implement ats run orchestrator and output folder") completed together.
**Decision**: Marked Efforts 06 and 07 as `done`. Verification: `cargo test --workspace` — 189 tests pass across 14 suites (pre-existing 183 + 3 new `pipeline_run.rs` tests + 2 new `pdf_smoke.rs` tests + 1 `ats-pdf` unit test); clippy clean under `-D warnings`. New surface in `ats-core`: `PdfWriter` / `PdfError` ports, `VecAuditSink`, `RunFolder::new_with_started_at`, `RunFolder::aggregated_token_usage`, `format_run_dir_ts`, and `pipeline::run(...)` returning `Result<(RunFolder, RunPipelineResult), RunPipelineError>` with an `Early` / `Late` split for AC-6.2 semantics. New crate impl: `Markdown2PdfWriter` (`markdown2pdf::parse_into_file` + `*.tmp`/rename atomic write, %PDF- magic guard, does not auto-create parent dirs so missing parents surface as exit 7). New CLI handlers: `commands/pdf.rs` (`tokio::task::spawn_blocking`, records `bytes_written` + `output_path`) and `commands/run.rs` (full post-scrape wiring with a scrape-phase `VecAuditSink` + stages-phase `FileAuditSink::create_with_records` that flushes buffered scrape audit lines before keywords runs). `run.json` now includes `scrape_slug`, `cached`, `cached_baseline`, `cache_path`, `keyword_density`, `density`, `total_tokens`, `final_pdf`, plus the built-in `token_usage_total`. Tests added: `packages/ats-core/tests/pipeline_run.rs` (happy path / offline Early failure / keywords × 3 schema-invalid Late failure) and `packages/ats-cli/tests/pdf_smoke.rs` (magic-byte happy path / unwritable parent → exit 7).
**Rationale**: The `Early` / `Late` split cleanly models AC-6.2 ("pre-scrape failure → no run folder; post-scrape failure → folder with `outcome=failed`") without leaking tri-state into callers. The buffered `VecAuditSink` → `FileAuditSink::create_with_records` pair keeps the scrape HTTP audit line in `llm-audit.jsonl` even though the run directory doesn't exist yet at scrape time, satisfying NFC-19 for the orchestrator.
**Alternatives considered**: Building a dedicated `EphemeralRun` type separate from `RunFolder` (rejected — `RunFolder::new_with_started_at` plus an in-memory `VecAuditSink` captures everything the journal spec called for with less surface area); auto-creating the PDF output parent (rejected — would mask the AC-6.4 error path); folding `cached` and `cached_baseline` into one key (rejected — Effort 07 locks `cached_baseline`; we keep `cached` as a backward-compatible alias already consumed by `ats render`).
